# 竞品/市场调研 Procedure

本 procedure 用于把用户的竞品、市场、官网、定价、趋势或公开资料问题，转化为一组可执行的公开网页查询，并基于搜索候选和抽取到的页面证据形成可检查的总结。

核心不是“搜一次就回答”，而是：

1. 明确研究口径。
2. 改写出覆盖关键维度的查询。
3. 判断搜索结果是否值得下钻。
4. 抽取少量高价值页面作为证据。
5. 在信息足够时总结；不足时说明缺口和下一步查询方向。

本 procedure 只处理公开资料读取和分析。不创建外部业务记录、不联系外部对象、不发送通知、不提交表单，不处理登录后内容、付费墙、绕过访问限制、个人隐私、泄露文件或非公开商业秘密。不能把搜索摘要、营销文案或单一来源直接当作已验证事实。

## 可用 Connector

- `{@connectors.web.search}`：用于开放网页搜索和新闻主题搜索。它发现候选来源，不直接提供已验证事实。可通过 query、topic、timeRange、includeDomains、excludeDomains、country 等参数控制查询范围。
- `{@connectors.web.extractPages}`：用于抽取已选 URL 的页面正文。涉及价格、功能、客户、融资、发布时间、监管、市场规模、重要对比结论或需要引用的内容时，必须优先抽取页面原文后再总结。

## 最小输入

用户至少需要给出一个研究对象：公司名、产品名、网站、行业、赛道、市场主题、竞品名单或明确问题。

如果缺少研究目的、地区、时间范围或输出形式，但已有明确研究对象，不要逐项追问阻塞流程；先用默认口径推进，并在回复开头说明默认值。只有这些缺口会显著改变查询方向、来源筛选或结论粒度时，才需要追问。

需要追问时，不是每轮只问一个。应一次性问 1 到 3 个最影响查询质量的问题，并把其余字段用默认值处理。示例：

> 为了避免查询方向发散，请先确认 3 点：研究对象是哪个公司/产品/赛道？这次主要服务找竞品、比较定价定位，还是看市场趋势？时间范围按近 12 个月可以吗？

如果用户没有说明地区、语言、时间范围或输出形式，可以先按默认口径推进，但回复开头必须说明默认值：公开中文和英文资料、优先近 12 个月、先做概览型调研。涉及“最新”“最近”“当前”“今天”“本周”等表达时，查询必须显式加入近期新闻或时间范围。

## 查询改写原则

查询改写的目标是覆盖“对象、语义、来源、时间、地域、证据类型”六个维度，而不是简单重复用户原话。

每轮研究先生成 3 到 6 个互补查询，优先覆盖这些查询族：

1. 对象查询：公司、产品、官网、品牌、缩写、别名、英文名、中文名。
2. 品类查询：行业名、赛道名、问题域、用例、购买场景、替代方案。
3. 竞品查询：competitors、alternatives、vs、comparison、best tools、top platforms、reviews。
4. 证据查询：pricing、features、customers、case studies、docs、FAQ、press、blog、funding、launch、changelog。
5. 时间查询：latest、recent、today、this week、2026、last 12 months，并配合 `topic: "news"` 或 `timeRange`。
6. 权威来源查询：官网域名、公司公告、文档、定价页、监管/公告页、行业报告、可信媒体或目录站。

不要只把用户问题翻译成一个宽泛 query。宽查询用于发现候选，窄查询用于验证结论。

### 查询改写模板

竞品发现：

- `<目标产品> competitors alternatives <核心用例>`
- `<目标产品> vs <已知竞品或品类>`
- `best <品类/用例> tools for <目标用户>`
- `<行业/赛道> software vendors platforms`
- `<目标产品> reviews comparison pricing`

竞品画像：

- `<公司/产品> pricing features customers`
- `<公司/产品> positioning product docs`
- `<公司/产品> case studies customers`
- `<公司/产品> funding launch partnership recent news`
- 已知官网时，用 `includeDomains: ["<domain>"]` 搜索 `pricing features customers docs blog press`。

定价定位：

- `<产品> pricing plans`
- `<产品> enterprise pricing sales contact`
- `<产品 A> vs <产品 B> pricing features`
- `<产品> docs limits seats usage billing`
- 已知官网时优先限定官网域名找 pricing、plans、docs、FAQ。

市场趋势：

- `<赛道> market trend recent news`
- `<赛道> funding product launch partnership regulation`
- `<赛道> report 2026`
- `<核心技术/使用场景> adoption risk opportunity`
- 时间敏感问题必须使用 `topic: "news"`，并按用户表达设置 `timeRange`。

官网信号：

- `<domain 或品牌> pricing features customers case studies docs blog press careers`
- `<domain 或品牌> positioning customers`
- 对已知官网优先使用 `includeDomains` 限定域名；当前不做全站抓取，不声称完成站点遍历。

深度报告：

- 先拆成多个子问题分别查询：竞品格局、目标客户、定价、产品能力、近期动态、机会风险、证据缺口。
- 每个子问题都应有独立 query，不要用一个“大而全”的 query 期待得到完整报告。

## 搜索结果筛选

搜索结果只提供候选来源。每次搜索后先做来源分层，再决定是否抽取。

优先级从高到低：

1. 官方或一手来源：官网、定价页、文档、公告、博客、新闻稿、监管/备案/交易所披露。
2. 权威二手来源：可信媒体、行业报告、研究机构、公开数据库、主流目录/评测站。
3. 辅助来源：社区讨论、评论、列表页、SEO 聚合页、个人博客。

筛选时必须：

- 按域名去重，避免多个搜索结果来自同一站点造成假共识。
- 区分官网、新闻、报告、目录、评论、社区和营销页面。
- 优先保留有日期、作者/机构、明确页面标题和具体内容的来源。
- 对明显 SEO 聚合、低质量转载、无日期、标题党或只重复官网营销词的来源降权。
- 搜索结果互相矛盾时，不直接总结，必须下钻抽取或换查询验证。

## 何时需要下钻抽取

只要结论会影响用户判断，且搜索摘要不足以支撑，就需要 `{@connectors.web.extractPages}` 下钻。下钻是为了获得可引用、可核验的页面正文，不是为了收集更多噪音。

必须下钻的情况：

1. 要引用价格、套餐、功能限制、客户名单、发布时间、融资金额、市场规模、监管结论或产品能力。
2. 用户要求“证据链”“带来源”“可核验”“完整报告”“对比矩阵”。
3. 结论依赖某个官网页面、公告、报告或新闻正文。
4. 搜索结果只给摘要，无法判断候选是否真相关。
5. 搜索结果互相冲突，或同一事实在不同来源里日期、金额、范围不一致。
6. 要把某家公司列为直接竞品，而不是只列为可能相关。
7. 要判断“最新”“近期”“当前状态”，需要确认页面日期或公告时间。

可以不下钻的情况：

1. 用户只要非常粗的候选方向，且回复明确标注为“初筛候选”。
2. 搜索结果只用于生成下一轮查询词或候选列表，不直接形成事实结论。
3. 来源明显无关、重复、低质量，抽取不会提升判断质量。
4. 用户问题本身只需要建议如何开展调研，而不是输出事实结论。

## 下钻到什么程度算够

信息足够不是“搜完所有资料”，而是“已有证据能支撑当前回答粒度，并且主要不确定性已经标注”。

通用停止条件：

- 关键结论至少有一手来源，或一个一手来源加一个独立二手来源支撑。
- 高风险事实来自抽取正文，而不是搜索摘要。
- 同一维度继续搜索只返回重复来源、低质量来源或边际信息很低的来源。
- 用户要求的比较维度已有可回答内容；没有证据的格子明确标注“公开资料不足”。
- 主要反例或矛盾来源已检查，仍无法消除的差异被写进“不确定性”。

按任务类型的足够标准：

- 竞品清单：有 5 到 10 个候选；每个候选有相关性理由；能区分直接竞品、间接竞品、替代方案和相邻玩家；至少抽取高价值来源确认核心候选。
- 竞品画像：每个重点对象至少有官网/官方页面证据；关键判断尽量再有一个独立来源。缺少独立来源时必须降级为“官网自述”。
- 定价定位：必须优先抽取官方 pricing、plans、docs 或 FAQ。没有官方价格时，只能说“公开价格未确认/需联系销售”，不能用第三方传闻补表。
- 市场趋势：至少看到多个独立来源围绕同一趋势反复出现，或有一个权威报告/公告加若干新闻事件支持。不能把单篇文章当作行业趋势。
- 官网信号：抽取 3 到 5 个高价值公开页面通常足够做初步判断，例如首页、pricing、features、customers/case、docs/blog/press。不能声称完成全站分析。
- 深度报告：每个核心章节都有至少一个可核验来源；仍缺证据的章节作为假设或后续调研问题，不强行补全。

信息不足时不要继续盲搜。应说明目前缺的是对象、地区、时间、竞品名单、官网、权威来源还是具体维度，并给出下一轮建议查询。

## 如何总结

总结的目标是帮助用户决策，不是堆搜索结果。输出必须先说明研究口径，再按事实、判断、假设组织内容。

默认结构：

1. 研究口径：对象、地区/语言、时间范围、资料类型、主要限制。
2. 关键结论：只写对用户问题有用的 3 到 7 条。
3. 证据与来源：列出支撑结论的 URL、来源类型、日期或日期不可得。
4. 对比或分类：竞品、趋势、定价、官网信号按业务维度分组。
5. 不确定性：哪些结论证据弱、冲突、过期或缺少官方来源。
6. 下一步验证：下一轮应查什么、需要用户补什么、哪些页面值得进一步抽取。

表达规则：

- 用“公开资料显示”描述抽取正文直接支持的事实。
- 用“基于这些来源可初步判断”描述分析推断。
- 用“仍需验证”描述没有足够证据的假设。
- 不为了表格完整而填空；没有证据就写“公开资料不足”“未能确认”“需要官方页面或用户补充”。
- 不把搜索排名当作市场地位，不把官网营销话术当作事实，不把旧资料当作当前状态。
- 重要事实必须带来源线索；高风险事实优先引用原文来源或权威来源。

## 场景规则

### 竞品发现

先用宽查询发现候选，再用用例、目标客户、价格带、地区和公开资料质量筛选。候选可以进入列表，但只有经过来源确认的对象才能称为直接竞品。

总结时必须区分：

- 直接竞品：解决相似问题，面向相似用户或采购场景。
- 间接竞品：满足部分需求，但产品边界或目标用户不同。
- 替代方案：用户可以用来替代完成任务，但形态不同。
- 相邻玩家：相关但不能证明为竞品。

### 竞品画像

查询应围绕定位、目标客户、核心功能、定价、客户证明、渠道、近期动作。官网信息是“自述”，独立来源是“外部佐证”。如果只有官网信息，必须降低结论强度。

### 定价定位

优先查询并抽取官方 pricing、plans、docs、FAQ。无法找到官方价格时，不得使用二手来源填价格；可以列出“公开未确认/需要销售联系/可能地区相关”。

### 市场趋势

先按时间范围查询新闻和公告，再用行业报告或权威二手来源校验。总结必须说明时间窗口，避免把过期资料当成当前趋势。

### 官网信号

只分析公开可访问页面。通过域名限定查询找到首页、产品页、定价页、客户案例、文档、博客、新闻稿等高价值页面后抽取。当前不做站点遍历，不声称全站抓取。

### 深度报告

深度报告不是启动异步研究任务，而是把大问题拆成多个查询与抽取批次。每个章节都要说明证据强度；证据不足的章节用“待验证假设/后续问题”呈现。

## 缺失、冲突和失败处理

当最小信息不足时，先判断是否真的阻塞查询改写。缺少研究对象、网站、公司/产品、赛道或明确问题时必须追问；缺少目的、地区、时间范围或输出形式时，通常用默认值推进。确实需要追问时，一次性问 1 到 3 个最影响查询质量的问题，避免每轮只补一个字段导致低效。

当搜索结果不足时，按顺序尝试：

1. 扩大或缩小品类词。
2. 加入英文名、中文名、别名、缩写。
3. 加入 use case、目标用户、行业词。
4. 改用 competitors、alternatives、vs、reviews、pricing、docs、launch、funding 等证据词。
5. 增加时间范围、地区或新闻 topic。
6. 限定或排除域名，减少重复和低质量来源。

当抽取失败时，说明哪些 URL 不可访问或抽取失败，并优先寻找替代来源。不能编造页面内容。

当来源冲突时，优先官方和日期更新的来源；如果仍无法判断，明确写出冲突和需要用户或人工复核的点。

## 明确不支持

本 procedure 不支持获取非公开数据、登录后内容、付费数据库、个人隐私、泄露文件、内部客户名单、内部财务数据或商业秘密。

本 procedure 不支持代替律师、投资顾问、审计师、咨询公司、安全评估机构或监管专家作出最终判断。

本 procedure 不支持自动联系竞品、客户、媒体、供应商或投资机构；也不支持创建任务、发送消息、提交表单、下单、订阅服务或写入外部系统。

本 procedure 不支持要求系统伪造来源、隐藏不确定性、删除风险提示、绕过来源限制或把公开资料分析包装成内部权威结论。
